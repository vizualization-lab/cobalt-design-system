# Help, Feedback & Support FAQ

These questions cover where to go when you are blocked, want to report a problem, or need to give feedback on Cobalt.

## Where do I ask a question?

Start with [GitHub Discussions](%GITHUB_URL%/discussions) for open-ended questions, early idea checks, and “which path is right?” conversations.

If you already know you need structured follow-up, use the issue templates instead.

The [Contact](/resources/contact) page is the central index for these support channels.

## When should I open an Issue versus a Discussion?

Use **Issues** when you need an actionable tracked item:

- bug reports
- feature requests
- migration help
- structured support requests

Use **Discussions** when you need conversation first:

- early design feedback
- proposal shaping
- architecture questions
- onboarding questions that may not require code changes

## How do I request migration help?

Open a GitHub issue using the migration-help template and include:

- current framework and UI stack
- target Cobalt packages or docs
- the step you are blocked on
- what you already tried
- any timeline or rollout constraints

The contributor-facing details are summarized in [How to Contribute](/contributing/how-to-contribute).

## How do I report a bug or request a feature?

Use the GitHub issue templates:

- **Bug Report** for regressions or incorrect behavior
- **Feature Request** for additions or enhancements
- **General Support Request** when the issue is broader than a bug

Strong reports include exact package/version context, reproduction steps, and screenshots or minimal demos where possible.

## Which support channels are available today?

Available today:

- GitHub Issues
- GitHub Discussions

Planned but not yet fully launched:

- live chat
- office hours

The docs should treat Issues and Discussions as the real support channels today.

## What response times should I expect?

The current docs guidance is:

- GitHub Issues: about 1–2 business days
- GitHub Discussions: about 1–2 business days

Do not assume same-day live chat support unless that channel is formally launched and documented.

## How do I contribute feedback if I am not opening code right away?

Feedback does not need to start as a pull request.

Good entry points:

- GitHub Discussions for open-ended feedback
- GitHub Issues for specific bugs, requests, or migration blockers
- component or design proposal discussions when you are suggesting a larger change
